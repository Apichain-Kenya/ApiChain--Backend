## Project Overview

ApiChain Kenya is a blockchain-based organic honey traceability system for Kenya's
smallholder honey value chain. The backend provides the REST API that powers the
system's core functionality: farmer and aggregator onboarding, OTP-based authentication,
geo-spatial farm data management with PostGIS, and document verification.

The project is built by a team of 3. The three components are:

- **Blockchain** (ApiBlockchain repo) -- Solidity smart contracts, RBAC, state machine
- **Backend** (this repo) -- FastAPI REST API, PostgreSQL/PostGIS, JWT auth
- **Frontend** (Apichain-Frontend repo) -- Vue.js web interface

This backend will eventually integrate with the blockchain layer to record honey batch
state transitions on-chain, and serves the frontend with RESTful endpoints.

**Repository:** `github.com/Apichain-Kenya/ApiChain--Backend`

## Architecture Decisions (Locked In)

### Layered Architecture
- **Routers** handle HTTP concerns (request parsing, response formatting)
- **Schemas** (Pydantic) validate input/output at API boundaries
- **Models** (SQLAlchemy) define database tables and relationships
- **No service layer** -- routers contain business logic directly (intentional for prototype scope)
- If a service layer is added later, move all database queries out of routers

### Authentication Strategy
- **JWT tokens** via `python-jose` with HS256 algorithm
- **bcrypt** password hashing via `passlib`
- **OAuth2PasswordBearer** scheme at `tokenUrl="/auth/login"`
- Token payload: `{sub: user_id, role: "farmer"|"aggregator", exp: expiry}`
- Default expiration: 30 minutes (configurable via `ACCESS_TOKEN_EXPIRE_MINUTES`)

### OTP Flow
- 6-digit random code stored in `otps` table with 5-minute expiry
- Currently returns OTP in response body (development mode -- must use Twilio SMS before deployment)
- Verification sets `is_verified=True` and `verification_status="verified"`

### Geospatial
- **PostGIS** `Geography(POINT)` type for farmer/aggregator locations
- **SRID 4326** (WGS84) -- standard GPS coordinate system
- Point format is `(longitude, latitude)` -- note the order
- **GiST indexes** on location columns for spatial query performance
- **Nominatim OpenStreetMap API** as geocoding fallback when coordinates not provided
- User-Agent header: `"AgriScanAI-App"` (required by Nominatim terms)

### File Uploads
- Local filesystem storage in `uploads/` directory (farmers/ and aggregators/ subdirectories)
- Named as `{user_id}_{original_filename}`
- No cloud storage yet -- planned migration to S3 or equivalent

### Database Migrations
- **Alembic is the only way to change the database schema**
- Never use `Base.metadata.create_all()` in production
- Always create a migration with `alembic revision --autogenerate -m "description"`

## API Endpoints

### Auth Router (`/auth`)

| Method | Path | Request | Response | Auth |
|--------|------|---------|----------|------|
| POST | `/auth/send-otp` | query: `phone` | `{otp: "123456"}` (dev mode) | None |
| POST | `/auth/verify-otp` | body: `{phone, code}` | `{message: "...verified"}` | None |
| POST | `/auth/login` | body: `{phone, password}` | `{access_token, token_type, role}` | None |

### Farmers Router (`/farmers`)

| Method | Path | Request | Response | Auth |
|--------|------|---------|----------|------|
| POST | `/farmers/register` | body: `{first_name, last_name, phone, email?, password}` | `{message}` | None |
| POST | `/farmers/farm-details/{farmer_id}` | body: `{address?, longitude, latitude, number_of_hives}` | `{message, lat, lon, address}` | None |
| POST | `/farmers/upload-document/{farmer_id}` | multipart: `file` + query: `doc_type` | `{message}` | None |

### Aggregators Router (`/aggregators`)

| Method | Path | Request | Response | Auth |
|--------|------|---------|----------|------|
| POST | `/aggregators/register` | body: `{business_name, phone, email, password}` | `{message}` | None |
| POST | `/aggregators/details/{agg_id}` | body: `{address?, longitude, latitude, farmers_count}` | `{message, lat, lon, address}` | None |
| POST | `/aggregators/upload-document/{aggregator_id}` | multipart: `file` + query: `doc_type` | `{message}` | None |

**Note:** ALL endpoints currently have Auth=None. JWT protection is not yet applied. See Known Issues #3.

## Authentication Flow

```
1. Register       POST /farmers/register or /aggregators/register
                  -> Creates account with bcrypt-hashed password
                  -> Returns success message

2. Send OTP       POST /auth/send-otp?phone=+254...
                  -> Generates 6-digit code, stores in otps table (5-min expiry)
                  -> Returns OTP in response (dev mode)

3. Verify OTP     POST /auth/verify-otp {phone, code}
                  -> Checks code + expiry
                  -> Sets is_verified=True, verification_status="verified"
                  -> Deletes OTP record

4. Login          POST /auth/login {phone, password}
                  -> Checks both farmers and aggregators tables
                  -> Returns JWT: {sub: user_id, role: "farmer"|"aggregator", exp}

5. Use Token      Authorization: Bearer <token>
                  -> (NOT YET IMPLEMENTED -- no endpoints check tokens)
```

## Key Design Principles (Do Not Deviate Without Review)

1. **Layered architecture** -- Routers handle HTTP, schemas validate data, models define
   storage. If adding a service layer, move ALL business logic there -- do not split
   logic between routers and services.

2. **PostGIS for all geospatial data** -- Always use SRID 4326 (WGS84). Point format
   is `(longitude, latitude)` -- never reverse the order. Use GiST indexes.

3. **Passwords never stored in plaintext** -- Always use bcrypt via passlib.
   Never log, return, or expose passwords in any response.

4. **JWT tokens carry role** -- Every token includes `{sub: user_id, role: "farmer"|"aggregator"}`.
   Future roles (admin, distributor, processor) must follow the same pattern and match
   the blockchain contract roles.

5. **Alembic is the only way to change the database schema** -- Never use
   `Base.metadata.create_all()` in production. Always create a migration.

6. **Environment variables for all secrets** -- DATABASE_URL, SECRET_KEY, Twilio
   credentials, and any API keys must come from `.env`, never hardcoded.

## Cross-Component Integration

### Backend <-> Blockchain
- **Not yet integrated.** The blockchain component (ApiBlockchain repo) defines a 6-state
  honey batch lifecycle: CREATED -> HARVESTED -> PROCESSED -> LAB_VERIFIED -> PACKAGED -> DISTRIBUTED.
- Backend will need to call smart contract functions via Web3.py when batch state changes.
- Blockchain roles (BEEKEEPER, PROCESSOR, ORACLE, DISTRIBUTOR, ADMIN) should align with
  backend user roles when that mapping is implemented.
- Backend stores mutable data (profiles, documents); blockchain stores immutable hashes.

### Backend <-> Frontend
- Frontend (Apichain-Frontend repo, Vue.js) consumes this API at `http://127.0.0.1:8000`.
- CORS middleware is now enabled (`allow_origins=["*"]` for dev). Must be restricted for production.
- Frontend will send JWT in `Authorization: Bearer <token>` header.
- API docs at `/docs` (Swagger UI) serve as the frontend integration reference.
